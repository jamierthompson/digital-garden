# R4 — Agent Collaboration & the `AGENTS.md` Entry Point

> Research note (cited). Scope: how a solo, **agent-driven** portfolio repo should
> structure collaboration with autonomous AI agents — the `AGENTS.md` standard,
> agent-friendly docs, multi-agent orchestration (research → drafts → debate →
> synthesis), keeping agents anchored to decisions, citing over remembering, and
> clean handoffs. Recommendations are tuned to **this** repo: see
> [`../../../decisions.md`](../../../decisions.md) (D1–D23),
> [`../../../architecture-plan.md`](../../../architecture-plan.md), and the existing
> [`../../../audit/`](../../../audit/) trail, which already practices five-lens →
> debate → synthesis.

---

## 1. The `AGENTS.md` open standard (ground truth)

`AGENTS.md` is an **open, vendor-neutral standard** — "a README for agents." It is
now stewarded by the **Agentic AI Foundation under the Linux Foundation**, and
emerged from OpenAI Codex, Amp, Google Jules, Cursor, and Factory. One file works
across 20+ tools (Codex, Cursor, Copilot, Jules, Aider, VS Code, Zed, Amp, …).
[agents.md](https://agents.md/)

Key facts (all from the spec):

- **It complements, does not replace, the README.** "README.md files are for humans."
  `AGENTS.md` holds "the extra, sometimes detailed context coding agents need: build
  steps, tests, and conventions" that would clutter a README.
  [agents.md](https://agents.md/)
- **Plain Markdown, no required fields.** Popular sections: _Project overview, Build
  and test commands, Code style, Testing instructions, Security considerations_, plus
  commit/PR conventions and gotchas. [agents.md](https://agents.md/)
- **Nesting = nearest file wins.** In a monorepo you "place another `AGENTS.md`
  inside each package. Agents automatically read the **nearest** file in the directory
  tree, so the closest one takes precedence." (OpenAI's own repo ships 88 nested
  files.) [agents.md](https://agents.md/) — directly relevant to this repo's
  `app + studio` pnpm workspace.

This repo's root `AGENTS.md` currently carries only the managed
`<!-- BEGIN:nextjs-agent-rules -->` block (the "This is NOT the Next.js you know"
warning). That block is **generated by `create-next-app` and must not be hand-edited
inside the markers** (per the Next.js guidance in the global instructions). House
content goes _outside_ the markers.

## 2. What actually makes agent docs work (empirical, not vibes)

The strongest evidence is an empirical study of real context files
([arXiv 2511.12884](https://arxiv.org/html/2511.12884v1)) plus practitioner spec
guides ([ASDLC](https://asdlc.io/practices/agents-md-spec/),
[morphllm](https://www.morphllm.com/agents-md-guide)):

- **Minimalism wins; bloat actively harms.** LLM-_generated_ context files **reduced**
  task success while raising inference cost **>20%**; "unnecessary requirements …
  actively harm agent performance." If a rule can be **enforced by a tool already in
  the repo, it must not live in the docs.** ([arXiv 2511.12884](https://arxiv.org/html/2511.12884v1))
- **Don't restate what CI/configs already encode.** Focus agent docs on
  "architectural decisions and team conventions tools cannot infer, rather than
  restating executable commands already codified in configuration files."
  ([arXiv 2511.12884](https://arxiv.org/html/2511.12884v1))
- **Median real files are short** (~485 words Claude Code / ~535 Copilot) and use a
  **shallow, templated hierarchy** (single H1, H2/H3 sections). Free-form prose drifts
  into "context debt" — opaque to humans, noisy for agents.
  ([arXiv 2511.12884](https://arxiv.org/html/2511.12884v1))
- **Security & performance are the documented blind spots** (each ~14.5% prevalence)
  — agents over-index on functionality. Worth an explicit note **here** given the
  `brandColor` 500-risk [D9] and Core-Web-Vitals goals.
  ([arXiv 2511.12884](https://arxiv.org/html/2511.12884v1))
- **Treat as living config, edited by small increments** — but watch "append-only
  degradation"; prune, don't just accrete. ([arXiv 2511.12884](https://arxiv.org/html/2511.12884v1))

## 3. Cite, don't remember (version-pinned grounding)

The repo's central failure mode is **stale model memory** about Next 16 / React 19.
The fix is the same pattern the project already uses: **point agents at version-matched
primary sources and forbid guessing.**

- Bundled, version-exact docs at `node_modules/next/dist/docs/` are the source of
  truth for framework behavior (the global Next.js guidance + root `AGENTS.md` already
  say this). Agents must **read them before writing framework code** — async request
  APIs, `proxy.ts` (not `middleware.ts`), removed `export const dynamic`,
  component-level static/dynamic under `cacheComponents`, the `@layer` trap.
- In-repo source of truth for _decisions_ is [`../../../decisions.md`](../../../decisions.md)
  (D1–D23) and `§N` of the plan. Every non-obvious claim in our docs should anchor to a
  `[D#]`, a `§N`, or an external URL — mirroring ADR practice where **accepted records
  are immutable** and changes ship as a _new superseding_ ADR, never an in-place edit.
  ([AWS Prescriptive Guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/architectural-decision-records/adr-process.html),
  [Azure Well-Architected](https://learn.microsoft.com/en-us/azure/well-architected/architect-role/architecture-decision-record))
- Grounding/citation also reduces hallucinated consensus — see §4.

## 4. Multi-agent orchestration: research → drafts → debate → synthesis

The repo's `audit/` already runs the high-value pattern; the literature backs it:

- **Adversarial debate raises accuracy.** Independent agents critiquing each other
  reduce hallucination and improve factual accuracy; gains are **strongest when agents
  are diverse** (different role prompts/seeds) and **critiques are grounded in explicit
  facts**, with a judge that rewards verifiable reasoning.
  ([beam.ai patterns](https://beam.ai/agentic-insights/multi-agent-orchestration-patterns-production),
  [arXiv 2510.04488](https://arxiv.org/pdf/2510.04488))
- **Orchestrator–worker + evaluator–optimizer** are the canonical Anthropic workflows:
  a lead decomposes and **synthesizes** worker output; an evaluator loops feedback
  against clear criteria.
  ([Anthropic — Building Effective Agents](https://www.anthropic.com/research/building-effective-agents))
- **Anthropic's multi-agent research system** is the production blueprint and matches
  our audit shape: a **LeadResearcher (Opus)** plans + persists the plan to memory,
  spawns **parallel subagents (Sonnet)** with explicit objectives/output-format/tool/
  boundary guidance, then a **CitationAgent re-attributes every claim**. Reported
  **+90.2%** over single-agent on their eval — at **~15× tokens**, so reserve it for
  **high-value, hard decisions** (architecture, not routine edits).
  ([Anthropic — Multi-Agent Research](https://www.anthropic.com/engineering/multi-agent-research-system))
- **Two named pitfalls to encode:** (1) **synthesis can hallucinate a consensus that
  isn't in the underlying results** — require explicit conflict resolution, not a
  smoothing summary ([alexewerlof](https://blog.alexewerlof.com/p/multi-agent-system-reliability)).
  (2) **Workers duplicate work when they can't see peers** — Anthropic's fix was
  _better task descriptions in the orchestrator's delegation prompt_, not more agents.
  ([Anthropic — Multi-Agent Research](https://www.anthropic.com/engineering/multi-agent-research-system))

## 5. Clean handoffs & context isolation (Claude Code specifics)

- **Subagents start with a fresh, isolated context** — they don't see history, prior
  reads, or invoked skills; the only channel is the delegation prompt and the returned
  summary. So delegation prompts must be **self-contained** and return a **dense
  digest, not raw output**. ([Claude Code — Subagents](https://code.claude.com/docs/en/sub-agents))
- **Isolate verbose work** (test runs, doc fetches, log crunching) in subagents so only
  the summary re-enters the lead's context. **3–5 concurrent is the everyday sweet
  spot**; per-agent model routing sends hard reasoning to Opus, bulk to cheaper models.
  ([Tembo](https://www.tembo.io/blog/claude-code-subagents),
  [MindStudio](https://www.mindstudio.ai/blog/claude-code-sub-agents-explained))
- **Decision rule:** _Skill teaches the how · Hook enforces the rule · Subagent
  isolates the work_ — use together.
  ([ofox.ai](https://ofox.ai/blog/claude-code-hooks-subagents-skills-complete-guide-2026/))
- For long efforts: summarize completed phases to **external memory** (the repo's docs
  themselves) before spawning fresh subagents — handoff continuity over context-window
  stuffing. ([Anthropic — Multi-Agent Research](https://www.anthropic.com/engineering/multi-agent-research-system))

## 6. Recommendations for THIS repo

**A. Make root `AGENTS.md` the lean entry point, not a manual.** Outside the managed
Next.js block, add a short "Working in this repo" section that **points** (does not
duplicate):

- Source-of-truth map: framework facts → `node_modules/next/dist/docs/`; decisions →
  `docs/decisions.md` `[D#]`; system model → `architecture-plan.md` `§N`; build order →
  `build-phases.md`; how-we-work → `docs/handbook/`.
- The five model-breaking Next-16/React-19 facts (async request APIs, `proxy.ts`,
  removed `dynamic`, component-level static/dynamic, the `@layer` trap, literal dynamic
  imports) — as a _checklist with a "verify in bundled docs" instruction_, not prose.
- Hard rules that are **also tool-enforced**, stated once with the command:
  `pnpm lint · lint:css · lint:keys · typecheck · test · format` and the boundary/
  isomorphism/`@layer`/key-drift gates. Lead with "CI is the bar; run these before
  committing," then trust the tools rather than re-explaining them.
- Conventions agents can't infer: pnpm-only, Conventional Commits, never commit to
  `main`, `@/*` alias, `interface` vs `type`, no `any`, reference-by-key + literal
  `() => import("@/projects/<slug>")`.

**B. Add a nested `studio/AGENTS.md`** so agents working in the Sanity workspace get
schema/TypeGen/stega-on-`brandColor` rules locally (nearest-file precedence).
[agents.md](https://agents.md/)

**C. Codify the orchestration pattern in the handbook** (`process/`): for
**architecture-class** decisions, run the audit shape — _N diverse independent drafts
→ adversarial debate → cited synthesis → ADR in `decisions.md`_. Require: distinct
role-lenses, fact-grounded critiques, an explicit conflict-resolution step (no smoothed
consensus), and a citation pass. Reserve it for high-value decisions (token cost). For
routine work, single-agent with subagent isolation for verbose subtasks.

**D. Encode handoff hygiene:** delegation prompts are self-contained and name the
source-of-truth files; subagents return **dense digests**; long work persists summaries
to the docs before re-spawning. Don't add agents to fix coordination — sharpen the
delegation prompt.

**E. Keep docs minimal and pruned.** Right-size ruthlessly (mirrors the plan's
"right-sized, not maximal"). Don't restate CI; flag the security/perf blind spots
explicitly given `brandColor` [D9] and CWV goals.
