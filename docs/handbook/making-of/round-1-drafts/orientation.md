# Orientation

**Read this first.** It is the map: what lives where, the mental model, the rules you cannot break, and a cold-start sequence for any task. It does **not** re-explain the system — for that, the architecture docs are the source of truth, and this doc points you to the right one.

Where this handbook and the architecture docs ever disagree, **`../decisions.md` wins** (it is the source of truth for deltas to the plan). Where any doc and the framework disagree, **the bundled Next docs win** (`node_modules/next/dist/docs/`) — your training data is stale on this stack (see [Golden rules](#golden-rules)).

## What this project is

A personal portfolio + digital garden. Each **project is a self-contained module** — its pages, its interactive experience, the components its essay embeds, and its scoped tokens — composed on a shared invariant foundation. Content + brand seeds live in **Sanity**; the site renders on **Next.js 16 / React 19** on Vercel. Phase 0 is complete; the build proceeds in risk-retirement phases (`../build-phases.md`).

The stack, verified — do not contradict it:

| Concern       | Choice                                                                                                                                           |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Framework     | Next.js **16.2.9**, App Router, Turbopack, **Cache Components on app-wide** [D11]                                                                |
| UI library    | React **19.2.4**                                                                                                                                 |
| Language      | TypeScript                                                                                                                                       |
| Styling       | CSS custom properties + CSS Modules, organized with `@layer`. **No Tailwind, no JSON tokens, no Style Dictionary**                               |
| Content       | Sanity, a **standalone Studio in `studio/`** (a pnpm workspace package, not a `/studio` route) [D23]; typed GROQ via TypeGen → `sanity.types.ts` |
| Testing       | Vitest + React Testing Library                                                                                                                   |
| Lint / format | ESLint (`eslint-config-next` + `eslint-plugin-boundaries`) + Prettier                                                                            |
| Hosting       | Vercel (full SSR/RSC). Package manager: **pnpm** (workspace: app + `studio/`)                                                                    |

## The mental model

Four ideas carry the whole architecture. Internalize the shape; read the cited section before you build against it.

1. **Modules, not a monolith** (§1, §4). Each project is a self-contained module under `src/projects/<slug>/`; genuinely shared parts live in plain shared `src/` modules. Dependencies point **projects → shared, never back**, and never project → project. This is lint-enforced (see Golden rules).

2. **Three-tier theming** (§3.1 [D1]). Only **brand color, font, and feel/geometry** vary per project. Everything else (spacing, motion, breakpoints, type-scale, z-index, semantic colors) is the **invariant foundation** at global `:root`. The brand ramp + font are **engine-scoped** per `[data-project]`; feel is a small scoped override. The **public token contract is the generic layer** (`--brand-*`, `--font-face`, `--space-*`) — `--logx-*` is a project-internal alias, never what a shared unit codes against [D2].

3. **The OKLCH engine** (§3.2) is the load-bearing, genuinely hard piece — a pure, **isomorphic** function `(brandColor, scheme) → tokenSet` in `src/lib/oklch/`. It bakes contrast-solved, gamut-mapped `oklch()` literals server-side [D3, D4, D6], is **scheme-aware** (one color → both light + dark via `light-dark()`) [D5], and is **defensive — never throws** [D9]. One engine, three consumers (theming layer, the playground project, `/work` card swatches).

4. **Reference-by-key + the `ProjectScope` keystone** (§4.2, §6). Sanity stores **keys** (`componentKey`, `fontKey`, `embedKey`, `brandColor`); code resolves them. `keys.ts` is the single source of truth; resolvers are typed `satisfies Record<Key,…>` (missing entry = compile error) and return a typed `NotFound`, never a bare lookup [D10]. **`ProjectScope`** is the one server component that takes a scope's `brandColor` + `fontKey`, emits the flash-free scoped `<style>` (both schemes) + the resolved font's `.variable` class, in the prerendered shell via `use cache`. It is the resolution keystone — everything beneath reads `var(--brand-*)` / `var(--font-face)`.

## Golden rules (non-negotiable)

These are the things that silently break this specific stack, or that the owner has standardized. Most are lint/CI-enforced — but know them so you don't fight the tools.

**This is not the Next.js in your training data — verify in `node_modules/next/dist/docs/` before writing framework code.** The model-breaking facts (all confirmed against the bundled docs; see [`./accessibility-and-performance.md`](./accessibility-and-performance.md), [`./security-and-ops.md`](./security-and-ops.md), and the research notes in `process/research/R1-*`):

- **Request APIs are async** — `cookies()`, `headers()`, `draftMode()`, and route `params` / `searchParams` are all `await`-able Promises.
- **`export const dynamic` / `force-static` are removed.** With Cache Components, static-vs-dynamic is a **component-level** concern decided by `use cache` placement and where request-time APIs are touched. A route is a **prerendered shell with dynamic holes** (PPR). Uncached data outside `<Suspense>` is a **build-time hard error** [D11].
- **`middleware.ts` is renamed `proxy.ts`** (Node runtime only — setting `runtime` throws).
- **The `@layer` trap** [D12]: Next does **not** auto-assign CSS Modules to a cascade layer, and an **unlayered** module outranks **every** `@layer` style. Every `*.module.css` must declare `@layer foundation|brand|project { … }` or stay strictly var-consuming. Enforced by `pnpm lint:css`.
- **Literal dynamic imports** [D21]: `() => import("@/projects/<slug>")` per key — **never** a templated `import(\`…/${slug}\`)` (defeats bundler static analysis).
- **Do not** put `server-only` / `client-only` on the OKLCH engine — it breaks isomorphism [D14].

**Conventions the owner standardized** (these don't get re-litigated per task):

- **pnpm only** — never npm/yarn; `pnpm dlx`, not `npx`. Commit `pnpm-lock.yaml` on dep changes (CI is `--frozen-lockfile`).
- **Never commit to `main`.** Always a descriptive `feature/…`, `fix/…`, `chore/…` branch. One task ≈ one commit. PR into main; CI green; delete the branch after merge. (Details: [`./git-and-pr-workflow.md`](./git-and-pr-workflow.md).)
- **Conventional Commits** (`feat`/`fix`/`docs`/`style`/`refactor`/`test`/`chore`). Note the asymmetry: branch `feature/…` pairs with commit `feat:`.
- **TypeScript discipline:** `@/*` alias not deep relative chains; `interface` for extensible object shapes, `type` for unions/intersections; **no `any`** (use `unknown`, then narrow); type params + returns.
- **Never hardcode secrets** — env vars only; keep `.env.example` current; `.env*` is gitignored. The Sanity **project ID / dataset are public** by design (plain env in `ci.yml`); a Sanity **token is secret**, server-side only, never `NEXT_PUBLIC_*`.
- **Before committing:** code runs, lint passes, `pnpm format` clean, diff reviewed (no debug logs, no commented-out code, no unrelated changes). The full Definition of Done is [`./definition-of-done.md`](./definition-of-done.md).

## Starting a task from cold

1. **Find your phase.** Open [`../build-phases.md`](../build-phases.md). Confirm what's done and what the next task is — the build is **risk-retirement ordered**, so work the sequence; don't jump ahead.
2. **Read the cited sections.** Each phase task cites `§N` (plan) and `[D#]` (decisions). Open the architecture-plan section and the ADR before writing code. The ADRs are **binding** and may override the plan.
3. **Verify framework facts.** If your task touches routing, caching, request APIs, CSS layers, fonts, or rendering, open the matching doc in `node_modules/next/dist/docs/` — do not trust memory.
4. **Branch.** `git checkout -b <type>/<kebab-description>` off `main` (`feature/…` for a `feat:` commit). Never work on `main`.
5. **Build the smallest slice = one commit.** Co-locate the test with its subject [D18]. Match existing patterns in `src/`.
6. **Run the full gate locally before pushing** (this is exactly what CI runs, in order):
   ```bash
   pnpm lint && pnpm lint:css && pnpm lint:keys && pnpm format:check && pnpm typecheck && pnpm test && pnpm build
   ```
   Fix formatting with `pnpm format` — never by hand.
7. **If you touched the Studio schema**, regenerate and commit the types — the easiest gate to trip [D23]:
   ```bash
   pnpm --filter studio typegen   # then commit the updated sanity.types.ts
   ```
8. **Open a PR into `main`.** One purpose per PR; description says what/why/testing. All tasks done before opening — no checklists with unfinished items. CI must be green to merge; delete the branch after.
